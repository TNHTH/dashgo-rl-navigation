# AI 软件开发全自动工作流 v5.1 (Auto-Naming Edition)

本手册专为**极简主义开发者**设计。**你只负责提需求，AI 负责起名字和写代码。**

核心机制：中央控制文件 (The Master File)

我们将项目的所有上下文（技术栈、接口、进度、文件命名映射表）保存在 project\_master.md 中。

## 📂 准备工作：创建“共享大脑”

在你的本地或编辑器中，新建一个文件 `project_master.md`。

**你可以手动创建，也可以通过下方的【阶段 1】让 AI 帮你自动生成第一版。**

## 🔁 标准工作循环 (The Dev Loop)

1. **更新 Master 文件**：运行【阶段 1】，告诉 AI 你要做的功能，让它规划文件名。
2. **开启新窗口**：不要在旧对话里纠缠，直接开个新 Chat。
3. **投喂上下文**：把 `project_master.md` 全文发给 AI。
4. **发送指令**：使用下方的专用提示词。

## 阶段 1：项目初始化 & 需求变更 (Project Genesis & Updates)

**场景**：

* **Case A (从零开始)** ：你只有一个模糊的想法（比如“写个自动抢票软件”）。
* **Case B (功能迭代)** ：你想增加新功能。

**操作**：

1. 开启新对话窗口。
2. 如果是 Case A：直接发送下方提示词。
   如果是 Case B：先发送当前的 project\_master.md，再发送下方提示词。

### 🚀 激活提示词

```
**Role**: 首席架构师 & 产品经理 (Master File Maintainer)。

**Task**:
我是项目负责人。请根据我的输入，撰写或更新项目的核心控制文件 `project_master.md`。

**Input Status**:
- 如果我提供了现有的 `project_master.md`，请基于它进行**增量更新**。
- 如果没有，请基于我的想法**从零创建**。

**My Request/Idea**:
【在此输入你的想法。例如：“写一个自动备份文件夹的EXE工具，要求界面简洁”】

**Naming Authority (重要)**:
- **你需要全权负责命名**：所有代码文件、类、变量、数据库表名由你根据最佳实践自动生成。
- **不要问我文件叫什么**，你决定，然后记录在 Master 文件中。

**Output Requirement**:
请输出一份**完整**的 markdown 格式的 `project_master.md` 文件内容。
文件结构必须包含：
1. **Global Strategy** (项目名, 技术栈)
2. **File Structure Map** (你规划的文件树，标明每个文件的作用，**这是后续开发命名的依据**)
3. **API Contract** (核心接口定义)
4. **Active Features** (功能列表)


```

 *(拿到 AI 回复后，直接复制内容，覆盖/保存到你的* *​`project_master.md`​* *文件中。)*

## 阶段 2：后端/逻辑开发 (Backend/Logic Task)

**场景**：实现具体的业务逻辑。

**操作**：

1. 开启新对话窗口。
2. 发送 **最新版** `project_master.md`。
3. 发送下方提示词。

### 🚀 激活提示词

```
**Role**: 高级后端/逻辑工程师。

**Context**: 已加载 `Project Master Context`。

**Task**: 请为我实现功能：【输入功能描述，如：文件夹监控逻辑】。

**Auto-Naming Protocol**:
- **查阅 Master 文件**：根据 Context 中的 `File Structure Map` 找到对应的文件名。
- **如果没找到**：请自行创建一个符合规范的专业文件名（如 `folder_monitor.py`）。
- **不要问我名字**，直接写代码。

**Constraint**:
- **严格一致性**：必须严格遵守 Master Context 中的接口定义。
- **孤岛模式**：引入所有必要的 Imports。

**Output**: 告诉我是哪个文件，并提供完整的代码。

```

## 阶段 3：前端/界面开发 (Frontend/UI Task)

**场景**：画页面。

**操作**：

1. 开启新对话窗口。
2. 发送 **最新版** `project_master.md`。
3. 发送下方提示词。

### 🚀 激活提示词

```
**Role**: 高级前端/UI工程师。

**Context**: 已加载 `Project Master Context`。

**Task**: 请实现界面：【输入界面描述，如：主设置面板】。

**Auto-Naming Protocol**:
- **查阅 Master 文件**：根据 Context 找到对应的组件/页面文件名。
- **如果没找到**：请自行决定最专业的文件名（如 `SettingsPanel.svelte` 或 `MainWindow.xaml`）。

**Strategy**:
1. **Mock First**: 确保无后端也能预览界面。
2. **Style**: 严格遵守 Context 中的 UI 规范。

**Output**: 告诉我是哪个文件，并提供完整代码。

```

## 阶段 4：纠错与Bug修复 (Fixing & Refactoring)

**场景**：代码报错。

**操作**：

1. 在当前的开发窗口中。
2. 发送报错信息。

### 🚀 激活提示词

```
**Role**: 调试专家。

**Error**: 【粘贴报错信息】

**Task**:
1. 分析原因。
2. **输出修正后的完整代码**。

```

## 阶段 5：打包与交付 (Build & Ship)

**场景**：代码写完了，你想生成最终的 `.exe` 或可执行程序。

**操作**：

1. 开启新对话窗口。
2. 发送 `project_master.md`。
3. 发送下方提示词。

### 🚀 激活提示词

```
**Role**: DevOps 构建工程师。

**Context**: 已加载 `Project Master Context`。

**Task**: 请指导我将项目打包成最终的可执行程序 (EXE/App)。

**Final Artifact Name**:
【在此输入你想要的最终程序名字，如：AutoBackupTool_v1.exe】
*(如果不填，请根据项目功能概括一个简洁的名字)*

**Action**:
1. 提供打包脚本（如 `pyinstaller` 命令, `electron-builder` 配置等）。
2. 告诉我在终端输入什么命令来生成最终文件。

```
